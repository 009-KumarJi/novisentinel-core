# 04 — OSS 101 for First-Time Maintainers

**Things you don't know yet that you need to know.**

You said you're new to OSS. This document covers what nobody tells you until you've already made the mistake. Read it now; come back when something specific happens.

---

## 1. What "open source" actually means

OSS is not "the source code is on GitHub." That's source-available. **Open source** means the code is distributed under a license approved by the [Open Source Initiative (OSI)](https://opensource.org/licenses) — Apache, MIT, BSD, GPL, MPL, etc.

The difference matters because:
- "Open source" gets you onto trusted lists (awesome-lists, OSS Insight, GitHub Trending).
- It signals you don't have a hidden commercial trap.
- It enables ecosystem use — LangChain can integrate against you without legal review.

Source-available licenses (BSL, SSPL, Elastic License v2) are **not** OSI-approved and you'll be politely uninvited from "OSS" lists / events. Sometimes that's the right tradeoff. For your first project, probably not — Apache 2.0 (what you have) is the right starting point.

---

## 2. Apache 2.0 — what you actually agreed to

You shipped under Apache 2.0. That means:

**Anyone can:**
- Copy your code into a closed-source product.
- Sell a competing service based on your code.
- Modify it and ship a fork.
- Use it commercially without paying you anything.

**They have to:**
- Include a copy of the LICENSE file in any redistribution.
- Note that the original copyright is yours.
- State the changes they made (if any).
- Not sue you for patent infringement on the parts you contributed.

**You give up:**
- Any right to control downstream use.
- Patent rights on code you contributed (Apache 2.0 has an automatic patent grant).

**You retain:**
- Trademark rights on the name "NoviSentinel" — important! See §5.
- Copyright on your code (anyone using it must attribute you).
- Decision-making authority on the project (until you grant commit rights to others).

The patent grant is the main thing Apache 2.0 has over MIT. It's a meaningful protection. Keep Apache.

## 3. Things you are now obligated to do

You're not legally obligated to do any of this — Apache 2.0 explicitly disclaims warranty. But socially / reputationally, the community expects:

### Within 24 hours
- Respond to new issues (even just "saw it, looking soon").
- Acknowledge serious security disclosures.

### Within 7 days
- Triage every issue: bug / feature / question / wontfix / duplicate. Label them. Close obvious dupes politely.
- Review PRs — at minimum, comment on what's blocking merge.

### Within 30 days
- Ship a release that includes user-reported fixes.
- Update CHANGELOG.md.
- Respond to questions on Discussions / Discord.

### Ongoing
- Keep the README accurate. Outdated install instructions kill more projects than bugs do.
- Keep dependencies current. `pip-audit` and `npm audit` should be green.
- Tag versions with semantic versioning (MAJOR.MINOR.PATCH).

If you can't keep up with this volume, archive the repo or hand it off. **Leaving an OSS project to rot publicly is worse than never starting one.**

## 4. Security disclosures

This is the one thing that can cost you legally, not just reputationally. Pay attention.

### The pattern

Someone finds a vulnerability in your code. They email `security@novisentinel.dev` (per your SECURITY.md). You have to:

1. **Acknowledge within 48 hours.** Even just "I see this, working on it." Silence = the reporter goes public.
2. **Fix it within 30–90 days.** Industry standard is 90 days; security researchers will often grant extensions if you communicate.
3. **Coordinate disclosure.** Pick a date when you ship the fix. Tell the reporter when you're going public. Don't surprise them.
4. **Credit the reporter** in the release notes (unless they ask not to be named).
5. **File a CVE if material.** GitHub makes this easy via Security Advisories.

### What goes wrong

- You're busy and the disclosure email sits unread for 2 weeks. The researcher tweets about it on day 15. Your users find out from Twitter, not from you.
- You patch silently without telling users to update. The next major incident traces back to the unpatched version.
- You argue with the reporter about whether it's "really" a vulnerability. Don't. If a reasonable security researcher thinks it's a vuln, treat it as one.

### What to set up now

- A `security@novisentinel.dev` email that you actually check daily.
- A pinned GitHub Security Advisory draft template.
- A note in CONTRIBUTING.md: "For security issues, do not open a public issue — email security@."

## 5. Trademark — protect the name

Apache 2.0 gives away the code. It does **not** give away the name. This matters.

If a malicious fork ships and calls itself "NoviSentinel Cloud" or "Real NoviSentinel," you have a problem. The fix is **trademark registration.**

### What to do (in priority order)

1. **Right now:** add a `TRADEMARK.md` file declaring that "NoviSentinel" and the logo are trademarks of you/your entity, that derivatives must not use the name. Doesn't have legal force, but it sets the expectation.
2. **Within 6 months:** register the trademark in your country of operation. Costs $200–500 in India, $250–400 in the US. Use a lawyer or LegalZoom for the first one; once you've done it, the renewal is easy.
3. **If you go international:** Madrid Protocol filing covers ~80 countries for ~$1000. Save for after first revenue.

### What you cannot stop

- Anyone using the name "NoviSentinel" in a blog post, comparison, or article. That's fair use.
- A fork calling itself something derived ("my-novisentinel-fork") in a personal repo.

### What you can stop (with the trademark)

- A competing commercial product called "NoviSentinel."
- Confusing use of your logo.
- A trademarked subdomain like `novisentinel-cloud.com` selling your competitor's service.

## 6. Governance — how decisions get made

For now, you are the **benevolent dictator for life (BDFL)**. You make every decision. That's fine and standard for solo OSS projects in year 1.

What it looks like:
- All PRs require your approval before merge.
- All releases come from your machine / your account.
- Roadmap is your call.
- Disputes you resolve.

When does this need to change? Roughly:

- **<10 active contributors:** stay BDFL. No governance overhead needed.
- **10–50 active contributors:** appoint 1–2 "maintainers" with commit access. Add a `MAINTAINERS.md`. Decisions still mostly yours, but they can merge straightforward PRs.
- **50–200 active contributors:** form a maintainer team (3–5 people). Decisions by rough consensus, you have veto. This is roughly where Sentry and PostHog are.
- **200+:** formal governance model (steering committee, BDFL-with-bypass, foundation). Don't think about this until you have it.

Common mistake: pre-building governance before you have a community. You don't need a Code of Conduct enforcement procedure when you have zero issues open. You have a CoC (`CODE_OF_CONDUCT.md` — review it), and that's enough until something happens.

## 7. Dual licensing — the trick that makes open-core work

You're going to need this eventually. Here's how it works.

The enterprise product (`novisentinel-be`) wants to ship the OSS detectors. But if `novisentinel-be` is closed-source and uses Apache code, that's fine (Apache allows it). What if `novisentinel-be` is closed-source and uses GPL code? Forbidden.

So: **keep `novisentinel-core` Apache 2.0** so you can freely use it in the closed enterprise BE.

But also: **require contributors to the OSS to sign a CLA (Contributor License Agreement)**. The CLA gives *you* a license to also distribute their contributions under a non-OSS license if you ever need to (e.g. to ship a closed-source binary that doesn't need attribution).

You don't need this yet (you have no external contributors). Add a CLA before the first significant external PR is merged. Use [CLA Assistant](https://cla-assistant.io/) — it's free and integrates with GitHub.

The CLA conversation is mildly contentious in OSS. Some contributors refuse. That's fine — let them. The protection it gives you is worth the occasional lost contribution.

## 8. The maintainer's burnout cycle

Most OSS projects die in year 2. The pattern:

```
Month 1: Launch. Energy high. 200 stars. 20 issues opened in 48h. You answer all of them.
Month 2: New issues slow. You ship features. Still energized.
Month 3: First feature request you don't agree with. You push back, conflict ensues.
Month 6: You've answered the "how do I install this on Windows?" question 30 times. You start to resent users.
Month 9: Inbox of 50 unanswered issues. You feel guilty every time you open GitHub.
Month 12: You haven't shipped in 6 weeks. Comments accumulate: "is this project dead?"
Month 14: You archive the repo. Apologize publicly. Move on.
```

You will not believe this can happen to you. It will.

### How to avoid it

- **Set boundaries early.** "I respond to issues on Mondays and Fridays." Put it in CONTRIBUTING.md. People will respect it.
- **Triage aggressively.** `wontfix` is a valid label. Use it. Closing an issue you don't want to do is not rude — leaving it open and ignored is rude.
- **Recruit maintainers before you need them.** When you see a contributor doing thoughtful work, ask them to join. Even 1 extra maintainer halves the load.
- **Take vacation visibly.** Pin an issue: "Out for 2 weeks. Will reply on [date]." Better than disappearing.
- **Don't apologize for shipping slowly.** Just ship slowly. The community adjusts.

### The signal that you're burning out

- Opening GitHub feels like a chore.
- You're answering questions with shorter and shorter replies.
- You start drafting a "I need a break" post in your head.

When any of those start, **immediately** take 2 weeks off. Don't push through. The damage of a public flameout is much greater than the damage of a quiet pause.

## 9. Issue and PR triage — the boring discipline

This is the unsung 50% of OSS work. Spend an hour every Monday morning on it.

### Issue labels you should have

- `bug` — something broken
- `feature` — request for new capability
- `question` — user asking how to do something (move to Discussions if possible)
- `good first issue` — easy, well-scoped, for new contributors
- `help wanted` — you'd love a PR for this
- `won't fix` — explicit rejection (with reason)
- `duplicate` — link to the canonical issue
- `needs more info` — waiting on reporter
- `dependencies` — Dependabot / Renovate stuff
- `security` — usually closed and moved to private Security Advisory

### PR triage

For each open PR ask:
- Does the change match the project's direction? (If no: close politely with explanation.)
- Are there tests? (If no: ask for them.)
- Is the diff small enough to review in 30 min? (If no: ask to split.)
- Does CI pass? (If no: ask author to fix before review.)

Aim for: **every open PR has a comment from you within 7 days.** Closed-without-comment PRs are the worst signal in OSS. Even a "thanks but this doesn't fit the scope" is much better than silence.

## 10. The economics — can OSS pay rent?

Realistically, for `novisentinel-core` in isolation: **no.** OSS code by itself almost never generates direct revenue at the indie scale.

Channels OSS founders use to monetize, in rough order of likelihood:

| Channel | Likely revenue | Effort | Notes |
|---|---|---|---|
| **Open-core enterprise upsell** | $0–$1M/yr | high | What you're doing. Best fit. |
| **Hosted service (commercial)** | $0–$10M/yr | very high | Same as above — the enterprise BE *is* this. |
| **Support contracts** | $0–$100k/yr | medium | Charge for response time. Easy money once you have customers, but needs customers first. |
| **Sponsorships (GitHub Sponsors, Open Collective)** | $50–$5k/month | low | Realistic for personal pocket money, not for runway. |
| **Consulting** | $5k–$50k/engagement | high (1-on-1) | Hard to scale, kills OSS work. Avoid unless you need cash. |
| **Bounties / paid features** | $1k–$50k/yr | medium | Customers pay you to prioritize their feature. Honest, sustainable. |
| **Courses / books** | $1k–$20k/yr | very high | Only if you have a strong personal brand. |
| **Funded by employer** | $0–$200k/yr | varies | You build OSS on company time. Anthropic, Vercel, etc. do this. |

For you: **the path is open-core**. The OSS exists to make the enterprise sell. Don't pursue the lower-yield monetization paths (sponsorships, consulting) early — they're distractions.

## 11. AI / LLM-specific OSS gotchas

A few things that are unusually relevant for an AI-safety project:

### Model weights

- The **DeBERTa-v3 prompt-injection classifier** you use (protectai/deberta-v3-base-prompt-injection-v2) is itself open under Apache 2.0. Good — no licensing entanglement.
- **Detoxify** is under MIT. Also fine.
- **Presidio** is under MIT. Also fine.

If you ever fine-tune a model and want to ship the weights: most fine-tuned weights on HuggingFace use the base model's license. Read carefully — some base models (LLaMA, Mistral) have non-commercial clauses. **Don't ship a derivative trained on a non-commercial-clause base.**

### Reproducibility

OSS users will ask "how do you know the injection detector catches X?" If you don't have published benchmarks, they'll be skeptical. Plan for: a small public eval corpus with documented precision/recall numbers. Aim for this in months 2–3.

### Safety vs. capability

There's a small but real risk that releasing a detector publicly helps attackers learn to evade it. Most security researchers conclude **transparency wins** — closed detectors get bypassed at the same rate but without the community visibility. Mention this position in your README so curious people don't ask repeatedly.

## 13. The one question you should ask yourself every month

> **"If I quit working on this tomorrow, would anyone notice?"**

The answer:
- **Month 1:** no one. (Expected. You haven't launched.)
- **Month 3:** maybe 5 people would notice. (Some early users.)
- **Month 6:** if no one would notice, you have a graveyard project. Either re-invest seriously or archive.
- **Month 12:** if dozens would notice, you have a real OSS project.
- **Month 18:** if hundreds would notice, you have a community.
- **Month 24:** if companies depend on it, you have leverage.

The trajectory matters more than any one data point. If month 6 is the same as month 3, something's wrong with positioning, distribution, or the product itself. Don't be afraid to *pivot* the OSS project — what you launch as v1 doesn't have to be what you commit to forever.

## 14. Resources worth reading

The OSS founder canon. Skim these as you go; you don't need them all today.

- **"The Cathedral and the Bazaar"** — Eric Raymond. Old but foundational. Read the first 4 chapters.
- **"Working in Public"** — Nadia Asparouhova. The best modern book on what OSS maintainers actually do.
- **Karl Fogel, "Producing Open Source Software"** — free online. Reference for governance, releases, contributor relations.
- **Sustain OSS podcast** — interviews with maintainers. Pick episodes that match your stage.
- **OpenCore Ventures / Heavybit / OSS Capital** — investor blogs that cover open-core specifically. Useful when the enterprise side starts taking shape.
- **Sentry's licensing journey** (blog posts ~2019, ~2023) — case study in why MIT didn't work for them and what they did instead.

## 15. The 5 things to internalize

If you remember nothing else from this folder:

1. **OSS distribution is a 12+ month commitment.** Don't measure week 4 and despair.
2. **Show up every week.** Boringly. Consistently. The community rewards presence above brilliance.
3. **The line between OSS and enterprise is your business model.** Get it right (see [`01-STRATEGY.md`](01-STRATEGY.md)).
4. **Burnout is the leading cause of OSS death.** Set boundaries early.
5. **You don't have to be perfect. You have to be present.**

Now go ship the v1.0 release tag and write the launch post.
